# 1. 概述

## 1.1 背景

当前项目使用 Casdoor 作为统一认证入口，但 Casdoor 下可能接入多个不同的 provider。不同 provider 返回的 JWT claims、userinfo 字段名、字段含义和完备度并不一致。现有实现将这些差异直接暴露给了登录回调、请求认证中间件和本地用户落库逻辑，导致系统内部 user 信息存在不稳定和不一致问题。

根据当前代码分析，系统内至少存在三套并行映射路径：

1. 登录回调中基于 `/api/userinfo` + access token claims 的映射
2. 请求中间件中直接解析 Bearer JWT 的映射
3. JWT 解析失败后 fallback `/api/userinfo` 的映射

这三条路径对 `sub`、`id`、`universal_id`、`preferred_username`、`name`、`picture`、`owner` 等字段的优先级和兼容方式并不一致，因此会造成同一用户在不同 provider 或不同请求路径下被解析成不同的系统内身份。

## 1.2 核心问题

当前问题主要表现为：

1. **用户身份不稳定**：同一自然用户在不同 provider 下可能无法命中同一条本地用户记录
2. **字段语义混用**：某些逻辑将 `id`、`sub`、`universal_id` 混合作为主身份字段使用
3. **资料字段不统一**：显示名、头像、组织字段在不同入口下可能为空或值不同
4. **返回结构不一致**：能查到本地 user 和 fallback 原始 Casdoor userinfo 时返回体风格不同
5. **扩展成本高**：新增 provider 时需要修改多处代码，且难以保证行为一致

## 1.3 方案目标

本提案目标如下：

| 编号 | 目标 | 说明 |
|------|------|------|
| G1 | 统一身份解析 | 所有认证入口都产出统一的标准化身份对象 |
| G2 | 稳定主身份键 | 引入稳定 `external_key`，避免重复用户 |
| G3 | provider 配置化 | 不同 provider 的字段差异通过配置处理 |
| G4 | 统一资料同步 | 显示名、邮箱、头像、组织等字段统一更新策略 |
| G5 | 平滑迁移 | 尽量兼容现有用户数据与现有登录流程 |

## 1.4 非目标

本提案不直接覆盖以下事项：

1. 不在本阶段引入完整的账号合并 UI
2. 不处理 Casdoor 本身 provider 配置的管理能力
3. 不引入跨系统用户主数据平台
4. 不改变当前 OAuth 登录基本流程

## 1.5 设计原则

本方案遵循以下原则：

1. **认证入口统一归一化**：所有来源都先转换成标准身份对象
2. **身份字段优先于资料字段**：用户唯一性判断必须独立于显示信息
3. **规则集中定义**：避免在 handler、middleware、service 各自写 fallback
4. **行为可观测**：身份选择过程必须可审计、可排查
5. **渐进式改造**：兼容旧字段，支持登录即迁移
